Read section 2 first for the big picture, then jump to whichever component you're integrating with.

1. What AmanahCX Is

AmanahCX is a multi-tenant SaaS platform that combines a CRM, a contact-centre (with an AI voice agent), sales & invoicing, analytics, and a field-staff mobile app — one system hosting many isolated customer organisations ("tenants"). A typical tenant is a bank, an insurer, or a services business that wants: customers to call in and get handled by an AI voice agent in Urdu/English, every enquiry to become a tracked ticket with an SLA clock, and their field/sales staff to work from a phone app.

As an integration partner, your work generally falls into one (or more) of three buckets:

  • Connect Nadia (voice) to a different CRM — a customer already has their own CRM (Salesforce, HubSpot, Dynamics, a bespoke system) and wants Nadia's voice capability without adopting AmanahCX's CRM. See Section 4.
  • Connect another application to AmanahCX — a softphone, a BI tool, an automation platform (Zapier-style), or a customer's existing systems need to read/write AmanahCX data or receive events from it. See Section 6.
  • Extend or customise AmanahCX itself for a specific customer deployment — new module, new field, new workflow. See Section 3 and Section 11.

AmanahCX is multi-tenant and multi-sector: every customer organisation is a fully isolated tenant (§3.2), and each tenant is provisioned against a sector template (banking, insurance, education, and others) — creating a tenant automatically populates the sector-specific fields its contacts, tickets, and deals need (e.g. Case Type / Transaction Reference / Regulatory Deadline for a banking tenant), with no manual field setup required (§4.17 of the Master Product Document). Every label in the system is a starting point, not a fixed name: a tenant admin can rename any role or responsibility title (e.g. "Line Manager" → "Territory Manager") and any field name to match how that organisation actually talks about its own work — the underlying permission model and data structure never change, only what they're called.

This flow was checked directly against the code (not just prior documentation) on 2026-07-18. Two real bugs were found and fixed as part of that check:
  1. In createComplaintFromStructured (packages/api/src/routes/voice-bot.ts), a Nadia-created ticket's ticket_type column was hardcoded to the literal 'complaint' regardless of the sales/inquiry/complaint classification computed for queue routing — so a sales enquiry captured by voice routed to the right queue but could never satisfy the ticket_type === 'sales' check that creates a pipeline deal on accept (§4.3). Fixed to persist the actual computed type.
  2. SLA breach escalation (runSlaWorker in tickets.ts) notified every manager/tenant_admin tenant-wide on breach, regardless of department — a Support ticket breach would also page Sales and Complaints managers. This broke the department-isolation model the rest of the system enforces via the reporting tree (lib/visibility.ts). Fixed with a new getEscalationTargets() helper: L1 now notifies only the assignee's own line manager, L2 only that manager's manager, walking users.manager_id the same way visibility checks already do elsewhere — falling back to tenant_admin only when an agent has no manager configured, so a breach is never silently unnoticed.
Two step descriptions above were also corrected to match real behavior: escalation (steps 9–10) now correctly reflects the assignee's own two-level manager chain, department-scoped, not a tenant-wide broadcast; contact matching (step 3) accepts any two of phone/NIC/email, not only pairs that include phone.

2. System Landscape

At the highest level, AmanahCX is three independently-deployable pieces talking over HTTP(S)/SIP — not one monolith. This separation is exactly what makes each piece reusable outside AmanahCX: the CRM API doesn't know or care whether a request came from the web app, the mobile app, Nadia, or a third-party integration — they all speak the same REST/GraphQL contract.

3.2 Multi-Tenancy & Security

  • Row-Level Security (RLS): every tenant's data lives in the same tables, isolated by a tenant_id column and Postgres RLS policies. The API sets app.tenant_id as a session variable at the start of each transaction (DatabaseClient.withTenant).
  • Non-superuser DB role is mandatory: the app connects as a restricted role (crm_app), never as postgres. A BYPASSRLS or SUPERUSER role silently ignores every RLS policy — this was a real production incident, fixed 2026-07-10. Any integration that touches DATABASE_URL must preserve this.
  • Tenant resolution on every request, in priority order: X-Tenant-Id header → subdomain → custom domain → JWT claim.

3.3 Access Control Model

Four layers, each answering a different question:

LayerQuestion it answersMechanism
1. EntitlementWhat has the tenant licensed?tenants.entitled_features / active_modules — set by Super Admin, checked via requireModule()
2. RolesWhat can this role do?Tenant admin builds roles (Admin/Manager/Agent/Viewer + custom + Policy Admin governance role) — checked via requireRole() / requireScope()
3. Record visibilityWhose records can this user see?Line-manager tree filter — agent sees own, line manager sees team, manager sees department
4. Separation of dutiesStructural guardrailse.g. tenant_admin is administrative-only, blocked from operational/billing routes
For any new integration, decide up front which of these four layers your integration's credentials should sit at — e.g. a softphone integration typically needs a scoped API key (layer 2), not a full admin JWT.

4. Nadia Voice Agent Architecture

Nadia is a self-hosted alternative to Retell AI / Vapi, purpose-built for the Pakistani market (Urdu/English, self-hosted STT, local TTS voice). It runs as its own Python service — services/nadia-voice-agent — completely independent of the AmanahCX codebase. It is deliberately CRM-agnostic: it never imports AmanahCX-specific code.

4.1 SIP + Voice Pipeline Diagram

Fig 4.1 — Nadia's SIP-to-speech pipeline and its connector boundary to whichever CRM is configured.

4.2 The Call Pipeline, Step by Step

1
SIP wiring (one-time, per phone number): setup_sip.py creates a LiveKit inbound SIP trunk for the carrier's number, plus a dispatch rule that puts every inbound call into its own room and dispatches the named agent nadia into it, carrying the tenant ID as room metadata.
2
Call arrives: caller dials the number → carrier delivers it over SIP to LiveKit Cloud → dispatch rule creates room call-xxxx and starts a Nadia worker process for it.
3
Settings load: the worker extracts the tenant ID from job metadata and calls get_config() on the CRM connector to fetch that tenant's bot name, tone, speaking rate, voice, system prompt, and guardrails — this is what makes Nadia's behaviour fully configurable per-tenant from the CRM's own admin screen, no redeploy needed.
4
Speech loop: Silero VAD detects when the caller stops talking → buffered audio goes to STT (self-hosted Whisper on capable hardware, or hosted OpenAI transcription on constrained hosts) → transcript goes to the gpt-4o-mini LLM with the tenant's system prompt → LLM either replies directly or calls a tool.
5
Tools: identify_caller (recognise a returning contact), check_knowledge_base (answer general questions without a ticket), raise_ticket (file a complaint/case — plays a spoken hold line while the ~2s API call is in flight).
6
Reply spoken back: LLM text → Uplift AI TTS → (optional ffmpeg time-stretch if speaking rate ≠ 1.0) → audio frames → back down the same room/SIP leg to the caller.
7
Guardrails running throughout: concurrency/capacity gate and minutes-allocation gate (checked right after connect — if exhausted, Nadia hands off with an urgent ticket + optional live SIP transfer instead of running normally); silence nudge (asks twice, then ends the call); hard max-call-duration cutoff.
8
Call ends: full transcript (+ recording URL if enabled) is reported back to the CRM via call_ended().

4.3 The CRM Connector Contract

This is the single most important thing to understand for integration work: Nadia's voice/AI code never talks to a specific CRM's API directly. Every CRM-facing call goes through one Python Protocol with exactly six methods, defined in services/nadia-voice-agent/src/crm_client.py:

class CRMConnector(Protocol):
    async def get_config(tenant_id) -> dict | None
    async def create_ticket(tenant_id, call_id, ticket: dict) -> dict
    async def search_knowledge_base(tenant_id, question) -> dict | None
    async def lookup_contact(tenant_id, phone, nic, email) -> dict
    async def call_started(tenant_id, call_id) -> dict      # concurrency/capacity check
    async def get_minutes_status(tenant_id) -> dict          # minutes-allocation check
    async def call_ended(tenant_id, call_id, transcript, recording_url) -> None

Today there is exactly one implementation, AmanahCXConnector, which calls AmanahCX's own REST endpoints under /api/v1/voice-bot/livekit/* with a bearer token (LIVEKIT_INGEST_SECRET). get_client() picks the active connector from the CRM_CONNECTOR env var.

4.4 Connecting Nadia to a Different CRM

To connect Nadia to Salesforce, HubSpot, Dynamics, or a bespoke CRM: write one new class implementing the six CRMConnector methods against that CRM's own API/webhooks, and register it in get_client(). Nothing in agent.py, the VAD/STT/LLM/TTS pipeline, the SIP setup, or the tool definitions needs to change.
MethodWhat the target CRM needs to expose
get_configReturn the tenant's bot settings (name, tone, voice, system prompt, guardrails). If the target CRM has no such concept yet, this can start as a static/env-driven config.
create_ticketCreate a case/ticket/lead record and return an ID Nadia can read back to the caller.
search_knowledge_baseOptional — return a direct-answer match for general questions. Can return None always if the target CRM has no KB.
lookup_contactSearch by phone/NIC/email, return a confidence-graded match so Nadia can (or cannot) greet a returning caller by name.
call_started / get_minutes_statusOptional gating hooks — can be stubbed to always return "not exhausted / not over capacity" if the target deployment doesn't need usage limits.
call_endedPersist the transcript/recording somewhere on the target CRM's side (case notes, call log, etc.).

Everything else — the SIP trunk, LiveKit dispatch rule, VAD/STT/LLM/TTS pipeline, guardrails, silence handling, live transfer — is entirely CRM-independent and is reused as-is.

Nadia currently lives inside the AmanahCX repo as a convenience, but architecturally it is already a standalone product (comparable to Retell AI / Vapi) — the connector pattern above is exactly what makes that possible without a rewrite.

4.5 Agent Builder — Reusable Templates & Auto-Provisioning (2026-07-18)

Before this, every tenant's bot was configured by hand-editing that tenant's voice_bot_configs row directly. Three additions make this reusable and (for licensed tenants) automatic:

  • Agent Templates (voice_bot_agent_templates, Super Admin → Agent Templates) — a reusable "recipe" (name, sector, company name, department, tone, character, language, guardrails, voice, call direction, bot engine) created once and assigned to any workspace. Assigning is a one-time copy into that tenant's voice_bot_configs row — not a live binding — so it stays independently editable afterward. Assigning also auto-grants the voice_bot module (+ the LiveKit provider feature) if the tenant wasn't already licensed for it.
  • Multi-source knowledge base at the Super Admin level — the same 3-way ingestion the tenant-side page already had (typed text, URL crawl-and-extract, PDF/DOCX upload with server-side text extraction) is now also available when Super Admin manages a tenant's KB directly.
  • Auto-provisioning on tenant creation — in POST /super-admin/tenants, if the new tenant is licensed for voice_bot, a working config is seeded immediately: a sector-matched template if one exists (falls back to a generic template, then to bare defaults), greeting auto-filled with the tenant's real name, and IVR menu segments derived from what was actually licensed — Inquiry/FAQ self-service is always included, Complaint appears if ticketing is licensed, Sales appears if sales is licensed.
All three were live-tested against disposable test tenants/templates, not real client workspaces, with test data deleted afterward — except one self-caught slip: the agent-template assign endpoint was first tested against the real Haier Electronics tenant, briefly overwriting its greeting/tone/ language/system prompt with test values before the mistake was caught and the original values restored.

5.3 Offline & On-Device AI

Works with no internet

  • Card scan: on-device ML Kit text recognition (Android/iOS) extracts name/title/company/email/phone from a photo — no network call.
  • Voice capture: on-device speech recognition, multilingual (English / Urdu / Punjabi), with an offline dictation parser as fallback.
  • Save: if offline, the record is written to a local queue and auto-synced the moment connectivity returns (NetInfo listener).

Upgrades when online

  • Card photo / raw dictation can be sent to the server, where Claude produces a cleaner structured extraction than the on-device pass.
  • Every save — offline or online — shows a confirm dialog summarising what will be pushed to the CRM, so the field worker verifies AI-extracted data before it's committed.

6. Connecting Other Applications

This section is for integrating a customer's other systems — a softphone, an existing CRM, a BI tool, or an automation platform — with AmanahCX itself (as opposed to Section 4, which is about connecting Nadia to a different CRM).

6.1 REST & GraphQL API

All CRM data is reachable at /api/v1/* (REST) and via GraphQL (Mercurius). Both share the same auth, tenant resolution, and access-control middleware described in 3.3. This is the primary integration surface — the web app, mobile app, and Nadia's connector all use it.

6.2 Outbound Webhooks

A tenant can subscribe an external endpoint to CRM events and receive a signed POST whenever they fire:

POST /api/v1/webhooks
{
  "name": "My Integration",
  "url": "https://partner-system.example.com/hooks/amanahcx",
  "events": ["deal.won", "contact.created", "ticket.created", "voice.call_completed"]
}

Deliveries are HMAC-SHA256 signed; the receiver verifies with the shared secret. Failed deliveries land in a dead-letter queue with a replay endpoint (GET /webhooks/dead-letter, POST /webhooks/dead-letter/:deliveryId/replay) — useful for building a reliable sync against a customer's other system.

6.3 API Keys (Inbound)

For a third-party system to call into AmanahCX programmatically (rather than receive events from it), it authenticates with a scoped API key instead of a user JWT:

Authorization: ApiKey crm_live_xxxxxxxxxxxxxxxx
X-Tenant-Id: <tenant-uuid>

Keys are scoped per resource/action (e.g. contacts:read, deals:write, voice:read) — issue the narrowest scope the integration actually needs. Managed at Settings → Integrations → API Keys (tenant admin only).

6.4 Telephony / Softphone Integration

Two distinct patterns exist depending on what you're building:

  • Voice bot number (Nadia): SIP trunk → LiveKit dispatch rule, as described in Section 4. This is for AI-handled inbound calls.
  • Human agent softphone: the connectors module (modules/connectors) has provider adapters (Twilio, Vonage today) for agent-facing calling — inbound webhook normalisation, outbound call initiation, and call-completed events into the CRM's activity/ticket timeline. A softphone product integrates the same way a new telephony provider adapter would: implement the provider webhook contract and register it as a connector, rather than hand-building CRM-side call logging.

6.5 Common Integration Patterns

You're building…Use…
A softphone for human agentsTelephony connector adapter (§6.4) + webhooks for call-completed events

7. Data Model & Auth Essentials

Auth mechanisms

JWTHuman users (web + mobile). Authorization: Bearer <token>
API KeyProgrammatic/integration access, scoped per resource:action
Bearer secretService-to-service — e.g. Nadia → CRM uses LIVEKIT_INGEST_SECRET

Key data-model facts

  • Every ticket links to a contact (auto-matched or auto-created on inbound).
  • A sales ticket converts to a pipeline deal on acceptance (idempotent, tickets.deal_id).
  • Per-channel communication consent (WhatsApp/SMS/Email) is append-only and enforced before outbound sends.
  • SLA policies match by priority + channel/department/tags; escalation is multi-step with % thresholds.

10. Self-Hosted Migration (VM + Domain)

This section is a ready-to-execute runbook for moving off Railway/ Vercel/Supabase onto infrastructure you control — a single VMware VM reachable by a domain name. Hand this to whoever provisions the VM (ops/infra) — steps 1–11 below are written to be followed in order, with the one non-negotiable called out explicitly at step 4 (recreating the restricted crm_app role so Row-Level Security keeps working).

10.1 Target Architecture Diagram

Fig 10.1 — collapsing four managed services into one VM behind your own domain; LiveKit's SIP/media plane can stay external or move in too.

10.2 Migration Runbook

1
Provision the VM. Ubuntu 22.04 LTS, reserved/static internal IP, a public IP (or NAT port-forward for 80/443). Sizing: 4 vCPU / 8GB RAM covers frontend + API + Postgres comfortably; if Nadia's self-hosted Whisper STT also runs on this box, budget significantly more CPU — Whisper is what maxed out Railway's small hosts in production (§9.3), which is why STT_PROVIDER=openai (hosted transcription) exists as a fallback for constrained hardware.
2
Install the base stack. Node.js 20 LTS, Python 3.11, PostgreSQL 16, Redis, Caddy (automatic HTTPS — the same pattern already used for Nadia's MinIO recording storage), and ffmpeg (Nadia's speaking-rate control depends on it, §4.2 — easy to miss since nothing fails loudly without it until a tenant sets a non-default speaking rate).
3
Clone the repo. GitHub stays the source of truth — git clone github.com/munirrazaa/AmanahCX onto the VM.
4
Migrate the database — the step that must not be shortcut. pg_dump the Supabase database → restore into local Postgres → recreate the restricted crm_app role with NOSUPERUSER NOBYPASSRLS and only the grants it needs (§3.2 — either attribute silently disables every Row-Level Security policy). Point DATABASE_URL at crm_app@localhost, never at a superuser connection.
5
Build and run the API. npm install && npm run build in packages/api → run the compiled dist/ under a systemd service on an internal port, proxied by Caddy at api.yourdomain.com.
6
Build and serve the frontend. npm run build (Vite) in packages/frontend → the static dist/ files are served directly by Caddy at app.yourdomain.com — no Node process needed, same as Vercel today, just served locally.
7
Deploy Nadia. Python venv, pip install -r requirements.txt, confirm ffmpeg is on PATH, set CRM_API_BASE_URL to the new internal API address, run src/agent.py as a systemd service with agent_name=nadia set (§9.3 — this exact value mismatch broke production once already). LIVEKIT_URL/API_KEY/API_SECRET keep pointing at LiveKit Cloud unless you also self-host livekit-server on this VM.
8
Point the domain at the VM. Create A records for app.yourdomain.com and api.yourdomain.com → the VM's public IP. Caddy auto-provisions Let's Encrypt certificates the moment DNS resolves — no manual certificate step.
9
Lock it down. Only open 80/443 publicly; keep Postgres bound to localhost (never expose 5432); SSH via key-based auth only; keep crm_app non-superuser as the one non-negotiable carried over from today's setup.
10
Replace the "push and it's live" convenience. Railway/Vercel's auto-deploy doesn't exist for free on a bare VM — the simplest equivalent is a small webhook receiver (or a GitHub Actions self-hosted runner) that runs git pull && npm run build && systemctl restart api (and rebuilds the frontend static files) on every push to main. This is the one piece of automation the team now owns and must consciously rebuild.
11
Back up. Nightly pg_dump cron to storage off the VM, plus a VMware-level snapshot before any major change — a safety net most managed PaaS platforms don't hand you directly.
