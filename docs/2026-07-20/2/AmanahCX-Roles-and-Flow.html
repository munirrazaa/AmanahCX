🔐Role Hierarchy Overview

AmanahCX uses a numeric rank system with Postgres RLS enforcing tenant isolation at the database level.

🏢

Tenant Admin

Rank 40 — Highest tenant-level role

Owns and configures the entire tenant workspace. Can hard-delete closed tickets. Cannot access other tenants or platform settings.

  • ✓Create, view, edit all tickets in tenant
  • ✓Hard delete tickets (closed tickets only)
  • ✓Manage agents, managers, viewers
  • ✓Configure SLA policies, queues, routing rules
  • ✓View all audit logs across the tenant
  • ✓Access all reports and analytics
  • ✓Manage contacts (create, merge, delete)
  • ~Change ticket priority — requires written governance reason
  • ✗Access other tenants' data
  • ✗Delete open or in-progress tickets
🤖

Voice Bot

Rank 20 (service account) — System actor

Automated service account. Accepts inbound calls, detects intent, creates tickets and contacts, triggers ACD routing. No UI access.

  • ✓Accept inbound calls and detect intent
  • ✓Create tickets and contacts
  • ✓Trigger automatic routing
  • ✓Write to ticket audit log
  • RRead routing rules and queue configs
  • ✗Edit or close tickets after hand-off
  • ✗View reports, manage users, delete records
👔

Manager

Rank 30 — Supervises agents and queues

Operational supervisor. Can edit all fields on any ticket, reassign agents in emergencies, and change SLA priority with mandatory reason.

  • ✓View all tickets in assigned queues
  • ✓Edit ticket fields at any stage
  • ✓Change ticket assignee (optional reason)
  • ~Change priority — mandatory written reason
  • ✓View full audit log for supervised tickets
  • ✗Delete tickets, manage user accounts
📋

Line Manager

Rank 30 variant — Team-scoped supervisor

Same rank as Manager but scoped to a specific team or queue. Cannot cross team boundaries.

  • ✓All Manager actions — scoped to their queue only
  • ~Change priority — mandatory reason required
  • ✗Cross-team ticket access
  • ✗User management, system config
🎧

Agent

Rank 20 — Frontline support staff

Handles assigned customer tickets. TAT clock starts on acceptance. Limited to own tickets.

  • ✓Accept assigned tickets (TAT starts)
  • ✓Update status, add notes, resolve
  • ownEdit ticket fields — own tickets only
  • ✗View other agents' tickets, change priority
  • ✗Delete any record
👁️

Viewer

Rank 10 — Read-only observer / Originator

Read-only access for customers and auditors. Can add notes only on tickets they originated.

  • RView tickets linked to them as originator
  • ✓Add notes on own originated tickets
  • ✗Edit fields, change status, see internal notes
  • ✗Delete, view reports, manage users

✅ D-P1 · "Pending" Ticket Status — RESOLVED

✅ D-P3 · Voice Bot Contact Update Scope — RESOLVED
ISO 20000 §8.3.5 — Bots should only append, not overwrite verified data
BEFORE (stated risk): The concern was that the voice bot could overwrite human-verified contact fields (name, NIC, email) if it misidentified a caller.

AFTER (verified safe): Code audit of voice-bot.ts confirms the bot never calls UPDATE contacts SET name = ... or updates any PII field on an existing contact. The flow is:
  • Bot identifies caller by phone → SELECT id FROM contacts WHERE phone = $1.
  • If found: uses the existing contact ID as-is. No field updates.
  • If not found: INSERT INTO contacts with the data the bot collected — creates a new record only.
  • Ticket is then linked to the contact via contact_id FK. No PII fields on existing contacts are ever overwritten by the bot.
Deviation is resolved by design — no code change was needed.
WasRisk flagged: bot appeared to have write access to contacts
NowVerified append-only: bot SELECTs existing contacts (no update) and only INSERTs new ones — ISO 20000 §8.3.5 compliant
🧪 Functionally tested 2026-07-01: Code audit of voice-bot.ts confirmed — only SELECT on existing contacts by phone, INSERT for new ones only, no UPDATE contacts statement anywhere. ✅ PASS

✅ G-F1 · Repeat-Caller Detection — RESOLVED
ITIL v4 Problem Management — Recurring incidents trigger Problem record
BEFORE: Every call created an independent ticket with no link to previous contacts from the same customer. Repeat callers were invisible to managers; root-cause analysis was impossible.

AFTER:
  • On every ticket creation, if the ticket has a linked contact (contact_id), the system counts how many tickets that contact has raised in the last 30 days.
  • If count ≥ 3: recurrence_flag = TRUE and recurrence_count = N are set on the new ticket.
  • All active managers and tenant_admin are notified in-app: "Contact has raised N tickets in the last 30 days. Root cause review recommended."
  • Migration 038 adds recurrence_flag (boolean) and recurrence_count (smallint) columns to tickets table.
WasNo detection — every ticket independent, repeat callers invisible to management

💼Sales End-to-End Information Flow

Inbound inquiry → lead qualification → product/quote → deal creation → follow-up → close/conversion. Parallel to the Complaint flow but driven by Sales dept rules.

1

Customer Calls In — Inbound Sales Inquiry

Customer dials the sales line. Telephony routes to AmanahCX. Voice Bot greets caller with sales IVR options (New Product, Renewal, Pricing, Talk to Agent).

Actor
Voice Bot (rank 20 service account)
Intent options
new_product_inquiry · renewal · pricing_query · general_sales
Data captured
Caller CLI, timestamp, selected IVR option, spoken product interest
System event
call_started + sales_ivr_option_selected logged
2

Lead Qualification — Bot Collects Pre-Sales Info

Bot asks qualifying questions (budget range, product interest, existing customer?) to pre-populate the lead record before the agent joins.

Captured fields
Product interest · Budget indicator · Existing customer Y/N · Preferred callback time
High-value signal
If caller states specific product name → intent confidence jumps → priority = high
NLP output
lead_score (1–5) based on intent confidence + field completeness
3

Contact Lookup / Lead Creation

System searches contacts by caller phone. Three paths:

A — Existing customer
Contact found. Purchase history + prior tickets loaded. Flagged as upsell opportunity.
B — Known lead (no deal yet)
Lead record found. Pre-existing interest noted. Agent sees full lead history.
C — New contact
New Contact created with phone + IVR-captured name. Lead status = new.
4

Ticket + Deal Created

Bot creates a Sales ticket linked to the contact. System also creates a Deal record at "Prospecting" stage.

Ticket
Type = sales_inquiry · Priority from lead_score · Queue = Sales dept
Deal auto-created
Stage: Prospecting · Source: inbound_call · Linked to contact + ticket
Deal fields
Product interest · Est. value (if stated) · Lead source · Owner = assigned agent
Audit
deal_created + ticket_created; actor = bot
5

ACD Routes to Sales Agent

ACD assigns to the next available Sales agent. Routing uses skills-based matching (product specialisation) or round-robin within the Sales queue.

Skills match
Agent tagged for product type (e.g. Insurance, Broadband) matches caller's interest
No agent available
Ticket queued. Bot offers callback slot. Manager alerted if queue exceeds 5 min.
Pre-loaded context
Agent screen pops with: contact name, lead score, product interest, prior tickets
6

Agent Accepts — TAT Clock Starts

Sales agent accepts the ticket. TAT clock starts. Status changes to in-progress. Deal stage moves to Contacted.

Deal stage updated
Prospecting → Contacted (auto on ticket accept)
Agent can see
Full contact history · Bot transcript · Lead qualification answers · Deal value
Agent actions
Update product interest · Set expected close date · Add deal value estimate
7

Proposal / Quote Sent

Agent presents pricing or sends a quote link. Deal stage advances to Proposal Sent.

Quote fields
Product · Price · Validity period · Special offer code (if applicable)
Deal stage
Contacted → Proposal Sent
Follow-up SLA
If no customer response in 48h → auto follow-up task created for agent
Originator (Viewer)
Gets notified: "Your enquiry is being processed — agent will follow up within 48h"
8

Negotiation / Objection Handling

Agent logs objections and manager can offer discount approval. All price changes require manager sign-off.

Discount request
Agent submits → Manager approves/declines → logged in deal audit trail
Manager scope
Line Manager sees all deals in team. Can reassign or co-assign.
Deal stage
Proposal Sent → Negotiation
9

Deal Won / Lost

Won path
Deal stage → Closed Won · Ticket resolved · Invoice auto-generated · Customer notified
Lost path
Reason required (price / competitor / timing) · Deal stage → Closed Lost · Ticket closed
TAT stop
Recorded at ticket resolution. Total sales cycle time = accept → close.
Post-close
Won deals trigger onboarding workflow. Lost deals enter re-engagement queue after 90 days.
10

Reporting & Analytics

Conversion rate
Won / total deals · By agent · By product · By source
Pipeline value
Sum of est. deal values by stage — live dashboard for managers
Lost reason analysis
Top 5 lost reasons surfaced weekly — feeds product/pricing decisions
TAT compliance
Avg sales cycle vs target · First response time · Follow-up adherence
